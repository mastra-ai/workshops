Thinking About Performance

Not every check needs an LLM

Six strategies to keep your processor pipelines fast and cost-effective.

Fast
Regex pre-filters
Run pattern matching first — PII, profanity, format validation. Costs $0, runs in microseconds. Only hit the LLM if regex passes.
.then(regexPreFilter) // μs
.parallel([topicGuard, moderation]) // LLM
Fast
Use fast safeguard models
Classification tasks don't need your main model. gpt-oss-safeguard-20b is purpose-built for safety checks — faster and cheaper.
model: 'openrouter/openai/gpt-oss-safeguard-20b'
Parallel
Run independent checks concurrently
Topic classification and content moderation don't depend on each other. .parallel() runs them at the same time — total latency = slowest check, not sum of all.
.parallel([topicGuard, piiGuard, moderation])
Smart
Route to cheaper models per step
processInputStep can swap models dynamically. Use your best model for the first reasoning step, then downgrade for tool result processing.
step 0 → gpt-5.2 (reasoning)
step 1+ → gpt-5-nano (tool results)
Smart
Only check what matters
.branch() makes processing conditional. Don't run an LLM drift check on every step — check every 5th step instead.
.branch([
  [ctx => ctx.stepNumber % 5 === 0, driftMonitor],
])
Stream
Stream-time early termination
processOutputStream can abort mid-generation. If the response exceeds a token budget or triggers a safety flag, stop immediately — don't wait for the full response.
if (state.tokens > MAX) abort('Budget exceeded');
The performance hierarchy
Regex (μs, $0) → Safeguard model (ms, ¢) → Parallel LLM checks (latency of slowest) → Full model only for generation
