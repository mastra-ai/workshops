# ExecPlan: Transformer Architecture — v1 Transformer Set

## Purpose

After this change, DeckForge's author can opt into six named visual components by dropping a single HTML-comment directive before a block in markdown: `card-grid`, `demo-card-grid`, `stat-grid`, `callout`, `tui-window`, and `mode-toggle`. Unknown transformer names and malformed input produce build errors with actionable messages. The renderer deduplicates transformer CSS and JS per slide, so two `tui-window` blocks on one slide emit one `<style>` tag, not two. The author can delete the `<!-- df: card-grid -->` comment and the block renders as a plain bullet list — full two-way opt-in.

This plan is where DeckForge gains its actual visual vocabulary. The foundation plan (IAM-79) built the pipeline; this plan fills it.

## Scope

### Files to create

One file per transformer:

- `src/transformers/card-grid.js` — consumes `bullet-list`. Validation: every item must start with `**bold text**` (a bold span prefix). On render: emits a grid container with one card per item, the bold prefix as the card title, and the remaining spans as the card body.
- `src/transformers/demo-card-grid.js` — consumes `bullet-list` with 3–6 items. On render: emits a numbered card grid. Numbers are derived from item order (1, 2, 3…). This is structural/ordinal decoration and is permitted under the idempotency rule.
- `src/transformers/stat-grid.js` — consumes `bullet-list`. Validation: every item must start with a numeric span (e.g. `**70%**`, `**$2.4M**`, `**3×**`). On render: emits a stat grid with the numeric as a large figure and the remaining text as the caption.
- `src/transformers/callout.js` — consumes `blockquote` or `paragraph`. Params: `color=accent|green|yellow|orange` (default `accent`). On render: emits a callout box with the appropriate color class.
- `src/transformers/tui-window.js` — consumes `code-block`. Params: `title=<string>` (default: the code-block's `lang`, or "terminal" if no lang). On render: emits a TUI-window frame around the code, reusing `.tui-window` and `.tui-header` classes from `theme/shared.css`.
- `src/transformers/mode-toggle.js` — consumes `nested-bullet-list`. Validation: at least 2 parent items, each with at least 1 child. On render: emits a tabbed mode toggle; parent item text is the tab label, children are the panel contents. Emits inline JS (wrapped in an IIFE) to handle tab switching. The JS is deduplicated by the renderer: one copy per slide regardless of how many `mode-toggle` instances appear.
- `src/transformers/index.js` — exports the six transformers as a single registry, merged with the passthrough registry from the foundation plan. Also exports `resolveTransformer(block)` which returns the directive-selected transformer or the passthrough fallback.

### Files to modify

- `src/renderer.js` — wire the new transformer registry into the resolver. Ensure CSS/JS dedup works across the six new transformers (the foundation plan tested this surface; extend the test to cover the real transformers). Implement the error UX: unknown transformer name or failed `validate()` in dev mode → render an error panel in place of the block; in build mode → throw, causing `bin/deckforge.js` to exit non-zero.
- `src/shells/standard.js` (and others as needed) — accommodate transformer-emitted `<style>` and `<script>` blocks. The foundation plan already does this via the renderer's dedup pass; the change here is only to verify and extend.
- `theme/shared.css` — add any per-transformer classes that need to be shared (`.callout-accent`, `.stat-grid`, `.mode-toggle-tab`, etc.). Transformer-specific keyframes stay inside the transformer's own `css` output per the design doc §3. **Do not dump all transformer styles into `shared.css`.** Only classes that are genuinely shared across transformers belong there.
- `theme/animations.css` (create if absent) — extract any shared keyframes that multiple transformers or shells use (e.g. `chipFloat`, `scatterIn`). This was flagged in design doc §7.

### Files to create (tests)

- `test/transformers/card-grid.test.js` — validation happy path, validation error (missing bold prefix), render snapshot.
- `test/transformers/demo-card-grid.test.js` — validation for 3-item, 6-item, 2-item (error), 7-item (error); render snapshot exercising numbering.
- `test/transformers/stat-grid.test.js` — validation happy path, validation error (non-numeric prefix), render snapshot with `%`, `$`, `×` stat formats.
- `test/transformers/callout.test.js` — render snapshots for all four colors; validation for invalid color param.

- `test/transformers/mode-toggle.test.js` — render snapshot exercising 2-mode and 3-mode toggles; validation for single-parent (error). Includes a test that the emitted JS is deduplicated when two `mode-toggle` blocks appear on one slide.

### Conventions (from `AGENTS.md`)

- Transformer files are kebab-case.
- Transformers are pure functions. No file I/O, no DOM, no sibling access.
- Transformers never synthesize content words. Numbering and structural decoration is OK.
- Inline JS in `render()` output must be wrapped in an IIFE to avoid polluting the global scope.
- CSS in `render()` output should be scoped via the `ctx.slideId` — the renderer uses the transformer `name` plus `slideId` as the dedup key, but if a transformer's CSS would clash across transformers on the same slide, scope it with a class prefix.
- Default to no comments; document the *why* only.
- Conventional commits with `Refs: IAM-79` in the body.

## Open Questions

- [ ] **`stagger-in` parameter placement.** The design doc §9 leans toward `stagger-in` being a parameter on every list-consuming transformer (`<!-- df: card-grid stagger-in -->`). v1 scope: accept it as a boolean param on `card-grid`, `demo-card-grid`, and `stat-grid`. When true, the transformer adds a `data-stagger-in` attribute and a class that `theme/shared.js`'s existing stagger helper can pick up. If this turns out to require new JS in the transformer output, flag it in the Decision Log and scope-cut to "card-grid only" for v1.
- [ ] **Callout default color.** Design doc says `accent` is the default. Confirm this matches the most common existing usage in the workshop decks; if not, change to whatever the majority use and record in Decision Log.
- [ ] **Mode-toggle tab switching JS.** The transformer emits inline JS. Decide: (a) each mode-toggle instance has its own IIFE with a unique id prefix, or (b) one shared IIFE per slide that looks up toggles by a common class. Lean (b) because it simplifies the dedup rule — one script tag per slide — but (a) is more straightforward. Make the call during implementation and record rationale.
